Idea 1 · Group detail page 🟡 medium

App Store + Play Store info with download counts

Recommended: build

Show App Store / Play Store links + a hardcoded download count (with a "last updated" date stamp) on each group's /g/<key> detail page. Already have apple-logo.svg and android-logo.svg in the repo — assets are ready.

Pros

  • Context: "1.2M downloads" instantly tells the user whether they're looking at a major app or a long-tail niche
  • Quick exit to the actual app — useful for designers researching live UX
  • Hardcoded counts are honest (with the "as of" date) — no live API costs, no rate limits, no broken integrations
  • Same data model fits both stores cleanly; admin updates monthly

Cons / things to think about

  • Maintenance: someone has to refresh the numbers periodically. Stale-but-dated is fine; stale-and-undated looks sloppy
  • Not every group is a public app (some are dApps, marketing pages, or internal tools) — UI needs to gracefully hide when there's no store link
  • Play Store + App Store download counts use different units (App Store doesn't publicly show downloads — only ratings/reviews). May need a footnote

Data model

  • groups add app_store_url, play_store_url
  • groups add app_store_rating (text, e.g. "4.7 ★ · 1.2M ratings")
  • groups add play_store_downloads (text, e.g. "150M+ downloads")
  • groups add store_stats_updated_at (date)
  • All optional — UI hides the section entirely when both URLs are NULL

UI affordances

  • component New <StoreLinkCard> — store icon, name, rating/downloads, "as of" date
  • layout Two cards side-by-side on desktop, stack on mobile
  • edit Admin can edit values from the existing Group Appearance edit modal
  • empty When only one store is set, that card spans full width
Why "Build"

Genuinely useful context, scales to every group in the catalogue, low maintenance once admin updates land. The "as-of" date keeps the numbers honest. Fits naturally above the Designer Picks row on the detail page.

Idea 2 · Lightbox crop interaction 🟡 medium-high

Dotted crop lines + "pieces detach" animation on confirm

Split the verdict: dotted = build, detach = trial

Two changes bundled together. The first is straightforward polish. The second is a delight feature that could feel slow if not done carefully.

Part A — Dotted crop guide lines

What this is

  • Replace the solid crop border + grid lines with dashed/dotted style
  • Matches how Photoshop / Lightroom / Figma render crop guides — instantly recognisable as "this is a transient cut line, not a real boundary"
  • Pure CSS: border-style: dashed on the crop overlay rectangle and the rule-of-thirds grid lines

My take

Build. 5-minute change, zero risk, immediately more familiar to anyone who's used a real photo editor. Should have been dashed from day one.

Part B — Pieces detach on confirm

Before crop (drawing) On confirm (animation, ~400ms) After ┌──────────────────────┐ ┌──────────────────────┐ ┌──────────┐ │ - - - - - - - - - - │ │ ↗ piece 1 flies off │ │ │ │ ┌────────────────┐ │ → │ ┌──────────┐ │ → │ kept │ │ │ kept area │ │ │ │ kept │ ↘ │ │ area │ │ └────────────────┘ │ │ ↙ ↘ │ │ │ │ - - - - - - - - - - │ │ piece 3 piece 4 │ └──────────┘ └──────────────────────┘ └──────────────────────┘ ⤷ outer regions fade + rotate + drop away

What it would take

  • technique Split the original image into 4 absolutely-positioned divs (top / bottom / left / right rims) each with the same image clipped via clip-path or overflow:hidden + offset
  • animation On confirm, each rim gets a unique CSS transform (translate + rotate + opacity) and animates over ~300-400ms with cubic-bezier(0.22, 1, 0.36, 1)
  • finalise When animation completes, swap to the actual cropped image source
  • a11y Respect prefers-reduced-motion — fall back to a simple fade

Risks

  • UX cost Adds 300-400ms to every crop confirm. For someone cropping 10 shots in a row, that's 3-4 seconds of waiting on theatre they've already seen
  • complexity The 4-piece split is non-trivial. Edge cases: crop near image edge (some pieces have zero area), portrait vs landscape, very small crops
  • first-impression vs repeat Delightful the first 3 times, friction the next 30
Recommendation

Ship Part A immediately — dotted lines are pure win.
Park Part B as a Friday-afternoon experiment with two non-negotiables: (1) skippable (hold shift to skip the animation), and (2) reduced-motion fallback. If it lands well, keep. If it slows people down on the second day, revert.
