C Hover-shuffling cards (Mobbin pattern)

Same card layout as B, but each card shows a single thumbnail that shuffles through samples on hover. Higher per-card visual density, encourages mouse-driven exploration. More implementation work.

Visual logic, since this is hover-driven and HTML can't easily show that frozen:
  • Card resting state: 1 sample thumbnail filling the card
  • Mouse enters card: thumbnail cross-fades to the next sample every ~600ms
  • Mouse leaves: cycle pauses on the current sample
  • Click: drills into the element's screenshots (same drill-in as B)

Add-on, not standalone — you'd build B first and then optionally layer the hover-shuffle in a follow-up PR. Worth doing if it actually helps you skim faster, but skip if you don't want the extra animation surface.

Decisions I'd love your call on before code lands

  1. Pick a concept: A (chip cloud), B (card grid), or C (B + hover-shuffle). My recommendation is B — visual card grid, with the option to layer C in a follow-up PR if hover-shuffle adds value.
  2. Nav entry name: "Elements" (recommended — short, ambiguous in a good way, fits next to "Catalogue"/"Videos"/"Links") OR "UI Elements" (more specific but tighter on space and slightly redundant since the catalogue itself is full of UI).
  3. Default sort: by count (most-used first) so the highest-value elements are top, OR alphabetical for predictability. The grid view supports both; pick the default.
  4. Card sample strip — which 4 samples? 4 most recent (always fresh), 4 most-diverse-by-group (a mix across apps so the strip reads as variety), or 4 random per page load? Diverse-by-group is most informative but slightly more compute.
  5. Drill-in URL pattern: /ui-elements/button (clean, shareable, matches /g/<group> already in use) OR /?ui_element=button (lands in the regular catalogue with that filter pre-applied). The first is more "library", the second is more "filter".
  6. Three kinds in one section, or three sections? Today the AI labels three things on each screenshot: UI elements, UX patterns, page types. Sub-tabs (Concept B) bundle them in one section. Alternative: three separate top-level entries in nav (UI Elements, UX Patterns, Page Types). Sub-tabs are cheaper; separate entries are more discoverable.

Once you've picked, I'll commit this mockup to the planning branch + scope the PR1 build (data flow, route, components).
