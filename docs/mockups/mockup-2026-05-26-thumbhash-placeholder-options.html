Thumb-hash placeholder — visualisation

Five rendering strategies for the same thumb-hash blob. Hit Replay on any card (or the global one) — the cached real image is forced to refetch so you can watch the placeholder → image handoff. To slow the network down further, open DevTools and throttle to Slow 3G.

Option A · baseline

Blur fill

Thumbhash filled to the image's rendered box, blurred 8px. The state shipped in the lightbox fix.

Cheap, shows shape · hard rectangle edges sit on the dark bg.

Option B · refinedRecommended

Blur + radial vignette

Same blur, plus a radial mask so the placeholder fades into the dark lightbox bg — no hard rectangle.

Premium, edges-don't-exist feel · slight loss of edge preview.

Option C · stylised

Pixelated crisp

Raw 32-pixel decode rendered with image-rendering: pixelated. Retro mosaic that reads as "obviously loading".

Distinctive, playful · clashes with a premium / muted theme.

Option D · refinedRecommended pair w/ B

Choreographed handoff

Placeholder animates from blur(10px) → blur(0) over ~220ms, then the real image cross-fades. The blur "resolves" into the photo.

Feels intentional · only visible when the image isn't cached.

Option E · minimal

Avg-colour flood + shimmer

Single muted colour (here a hand-picked stand-in for thumbhash's mean) with a slow horizontal shimmer sweep — closer to skeleton than preview.

Calmest, neutral · loses shape preview entirely.

In production the placeholder is decoded synchronously from the stored thumbhash bytes (no network), so it paints on the first frame. Here we approximate that by giving the placeholder a 32×22 image with the same picsum seed as the full image — same artistic effect, just a real fetch instead of an inline decode.
