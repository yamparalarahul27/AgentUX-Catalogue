Changelog page + feature intro sheets

Splitting "What's New" into two: the existing list view becomes a dedicated Changelog page, and the "What's New" surface becomes a per-feature intro sheet that introduces one new thing at a time when the user first encounters a build with new features.

What I understood

Changelog pageA dedicated /changelog route (or account-menu entry) that lists every release historically — same content shape as today's whats-new.json, just a full browse-by-date page
What's New (after this PR)One feature intro sheet per new item. Triggered on first session after a build that has un-seen items. User dismisses or steps through; each item only shown once
Data sourceSame whats-new.json — adds a tracking layer for "has this user seen entry X yet?" (localStorage, simplest)

How to pick

The four concepts trade off along two axes — how intrusive they are, and how guided they are.

Less intrusive, less guided → B (corner card)
More intrusive, less guided → A (full sheet)
Less intrusive, more guided → D (rail of cards)
More intrusive, more guided → C (inline spotlight)

If you're trying to make sure people actually see the change, lean A or C. If you trust users to discover and only want a gentle nudge, lean B or D.

Then we scope

  1. Which concept for the intro sheet (A / B / C / D — or a mix)?
  2. Trigger: show automatically on first session after a build with unseen items, or only when user clicks an "N new" badge in the account menu?
  3. Tracking: per-user (in Supabase) or per-device (localStorage)? Latter is simpler, former survives device changes.
