4 Key design decisions

Decisions I want your input on before any code lands. Each one shapes the algorithm + UX.

Need your call on:

  1. When does a token become a chip?
    (a) Auto-convert — if "weex" matches a known group exactly (or as a unique prefix), instantly chip it as the user types.
    (b) Always require Tab / click — show suggestion, user explicitly accepts to convert. Less magic, more control.
    My recommendation: (b). Auto-converting can feel jumpy and gets confusing when a token matches both a group and a flow.
  2. Ambiguity: if "Login" matches both a flow named "Login" and a group called "Login Inc.", what do we do?
    (a) Show both as separate suggestions — user picks which.
    (b) Show one based on confidence (e.g. exact-match wins, ties go to flow).
    My recommendation: (a). Cheap to render two suggestions; user keeps control.
  3. Stop words: the example has "screen in Weex" — the word "in" is grammatical glue. Do we ignore it in matching?
    (a) Strip a small list ("in", "of", "the", "on", "and") from the text-search bucket so they don't filter results.
    (b) Don't strip — match literally — risk: "screen in Weex" gets fewer results than "screen Weex".
    My recommendation: (a). The user is writing natural language, not a database query.
  4. Order of chips: do chips preserve the typed position, or do they all jump to the front?
    (a) Preserve position — chips appear where the user typed them: [Flow:Deposit] screen in [Group:Weex].
    (b) Hoist to front — all chips collect at the left; free text always trails: [Flow:Deposit][Group:Weex] screen in.
    My recommendation: (a). Reads more naturally; matches the user's mental model.
  5. Multiple of the same kind: can the user have two group chips? (e.g. [Group:Coinbase][Group:Weex] login).
    (a) Allow multiple chips, OR them together — show screenshots in Coinbase OR Weex matching "login".
    (b) Only one chip per kind — typing a second group replaces the first.
    My recommendation: (a). Multi-select is already how the regular toolbar group filter works; consistency wins.
  6. Scope of entity recognition: just groups + flows, or also platforms (web / mobile), themes (light / dark), OS (ios / android)?
    (a) Just groups + flows (v1) — ship the simple version, add others later.
    (b) All structured fields from day one — five entity types: group / flow / platform / theme / OS.
    My recommendation: (a). Each new entity type adds disambiguation complexity. Validate the pattern with the two highest-value types first.
