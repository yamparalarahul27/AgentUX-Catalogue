UI Element tag — promoted from annotations

Today the UI Element filter is populated by the Labelling Studio only (admin / ResearcherAI). Annotations live in a parallel universe — useful for pinpointing pieces of the screen, but invisible to the filter taxonomy. The proposal: let any annotator promote an area annotation into a UI Element tag, growing the taxonomy democratically.

No existing backlog item — this is net-new. Thinking through it before filing.

1 · The two concepts today

How annotations and UI Elements live separately

Two parallel systems on the same screenshot, no connection between them:

Annotations

  • Drawn by anyone with edit rights in the lightbox
  • Area (rectangle) or pin (point); has free-text label
  • Stored in the annotations table per screenshot
  • Used for locating pieces of UX — "look at this here"
"Sign-up modal" "CTA button" "Skeleton state"

UI Elements

  • Set in the Labelling Studio only (admin / ResearcherAI)
  • Stored at metadata.label.screen_analysis.ui_elements as a string array
  • Aggregated across the catalogue and exposed in the Filter sheet
  • Used for categorising the catalogue — "show me every screen that has a Bottom Nav"
Bottom Nav Asset Strip Order Book

2 · What you proposed

One annotation flow, two outcomes

User draws an area around an Asset Strip on the Binance Trade screen. In the composer, instead of just typing a free-text label and saving, they get a small toggle: "Tag as UI Element". When checked, the label name (with autocomplete from existing UI Elements) becomes part of both the annotation and the screenshot's UI Element list — instantly filterable.

3 · Three implementation paths

How to wire it up

Path What changes Trade-offs
A Recommended Two writes on save. Annotation table unchanged. When the toggle is checked, the save handler ALSO patches metadata.label.screen_analysis.ui_elements on the screenshot (de-duped). Existing filter query untouched. 🟢 Minimal schema work. Existing filter just works.
🟡 Two writes — needs transactional safety or "best-effort second write" with a small reconcile job.
B More work Annotations become the source of truth. Add is_ui_element + ui_element_name columns to annotations. Filter query reads from annotations joined to screenshots. Existing studio-set ui_elements get backfilled into annotations as "studio-imported" rows. 🟢 Single source of truth, no drift.
🔴 Schema migration + filter query rewrite + backfill of existing studio data. Bigger blast radius.
C Skip Database trigger keeps them in sync. Annotations get the flag (like B), but a Postgres trigger maintains the jsonb on the screenshot row, so reads stay fast and the filter doesn't change. 🟢 Best of both worlds at read time.
🔴 Triggers add hidden behaviour — debugging gets harder. Overkill for current data volume.

4 · Risk: taxonomy bloat

"Asset Strip", "asset strip", "AssetStrip", "Asset list"…

The biggest danger of democratising the taxonomy is that everyone names things slightly differently. A clean studio-curated list of 40 elements becomes a sprawl of 200 near-duplicates inside a month.

Safeguards in v1

  • Autocomplete first. When the user starts typing, suggest existing UI elements (already used elsewhere). Encourages reuse over creation.
  • Normalize on save. Trim, collapse whitespace, Title-Case the saved value so "asset strip" → "Asset Strip".
  • Soft cap. If a new value differs from an existing one by < 2 characters (Levenshtein), warn the user inline: "Did you mean Asset Strip?"

Admin tools (v2 backlog)

  • Merge tool: select two UI element names → rewrites all references to the canonical one
  • Promote/demote: admins can mark certain UI elements as "official" (shown in filter) vs "informal" (hidden until merged)
  • Studio integration: Labelling Studio sees user-tagged elements with provenance, can curate them in

5 · Verdict

Worth building — Path A, with the v1 safeguards

Recommendation

Build it as Path A (two writes, existing filter). Ship with the three safeguards (autocomplete, normalize, near-match warning). Defer the admin merge tools to a follow-up — only worth building once we have evidence the taxonomy is bloating. Pair this with the parked Edit comments work — both touch the lightbox composer area.

Size estimate: 🟡 medium — half-day for the toggle + save-side write + autocomplete; another quarter-day for normalise + near-match warning. Most of the time is going to be polish on the autocomplete UX, not the data plumbing.
