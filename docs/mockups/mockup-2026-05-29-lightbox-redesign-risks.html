Lightbox redesign — what could go wrong

The redesign you sketched isn't just a paint job — pieces of the lightbox move from one spot to another. Each of these moves could quietly break something if I'm not careful. Below, every "thing that moves" with a picture of what's at risk and how I plan to keep it safe.

What could break How I'll prevent it

The rule you set: this is a UI-only change. Nothing functional should break. If I see anything that worries me, I tell you first.

What could break

The title is an editable component — click it, type, blur to save. Moving it to a new spot in the DOM could disturb the "click outside to save" logic or where the text input shows up.

How I'll prevent it

I'll move the title's JSX as a single block, without touching its props or behavior. After the move, I'll click-rename in the dev preview to confirm save-on-blur still works before showing it to you.

2

The group chip (Valr, Coinbase…) moves and changes shape

It's also a link — clicking it goes to /g/<group>

Today

The group is rendered as a coloured chip in the top bar. Clicking it navigates to the group detail page.

After redesign

The group becomes inline text next to a small icon in the metadata row of the side panel.

What could break

The chip is a clickable link. If I rewrap its container or change its element type, the link could lose its click handler or its route navigation.

How I'll prevent it

The group is rendered by an existing component (CatalogueGroupLabel) with a linkTo prop. I'll lift that component as-is into the new metadata row — same component, same prop. After moving, I'll click it in the dev preview to confirm it still navigates to the group page.

3

The close (×) button moves from the top bar to the screenshot's top-left

Has to close the lightbox AND not get in the way of the image

Today

× sits in the top-right of the lightbox header.

After redesign

× sits as a floating button in the top-left corner of the screenshot area.

What could break

The × needs to (a) still close the lightbox when clicked, (b) appear above the image (not behind it), and (c) not interfere with any drag / right-click on the image itself.

How I'll prevent it

I'll keep the exact same close button (same onClick, same icon), just position it absolutely inside the screenshot's container with a layered background so it's clearly readable on any image. After moving, I'll click × on the dev preview to confirm it closes; then click on the image itself to confirm no interference.

What could break

If I center the empty-state container by changing the layout of the WHOLE comments area, the populated list (when you HAVE comments) could end up scrolling differently or rendering in the wrong place.

How I'll prevent it

I'll only centre when the comments array is empty — using a conditional class so the populated-list code path stays untouched. After the change, I'll test both: empty list (centered message) and populated list (normal scroll).

6

Mobile keeps its existing bottom-sheet layout

Desktop change must not leak into mobile

Mobile (phones, narrow tablets) uses a totally different lightbox layout — the bottom sheet from PR #184. The desktop redesign should not affect mobile at all.

What could break

If I write a new CSS rule that isn't scoped to desktop only, it could leak into the mobile bottom-sheet layout — same kind of bug we hit in PR #185 (the "leak" you're worried about).

How I'll prevent it

Every new CSS rule for the redesign goes inside a desktop-only block (@media (min-width: 721px)) or stays outside any mobile media query. I'll explicitly open the lightbox on a phone-sized window during testing and verify it looks identical to today.

What's NOT changing (no risk, no work)

  • Action button bar (Edit · Crop · Save · Copy · Comment · Annotate · Delete) — already grouped the way your image shows
  • Comment composer position (already pinned to the bottom)
  • Squircle corners from PR #190 — the new redesign keeps them
  • All button click handlers, all keyboard shortcuts, all state — completely untouched
  • Mobile layout — completely untouched
  • Crop mode, comment add/edit, annotation add/edit — all untouched
